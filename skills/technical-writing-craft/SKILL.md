# Technical Writing Craft

## Overview

Great technical writing doesn't deliver facts — it delivers a **new way of seeing**. The reader finishes with a framework that changes how they interpret things they already know. Facts serve the framework, not the other way around.

## Core Patterns

### 1. Framework-First：框架先行

先找到统一框架，再挂事实。不要堆砌信息。

**做法：**
- 在动笔前问："有没有一个框架，能让所有例子都成为它的实例？"
- 框架可以来自历史（控制论）、数学（P vs NP）、物理（熵）
- 框架选好后，每个细节都在验证框架，读者"越读越通透"

**反面：** 先罗列 A 做了什么、B 做了什么、C 做了什么 → 读者记信息，不长见识

---

### 2. 例子的节奏：由远及近，逐步扩射

三个例子的排列顺序是精心设计的。

| 位置 | 特征 | 作用 |
|------|------|------|
| 第一个 | 遥远、具象、人人能懂 | 建立直觉，零抵抗 |
| 第二个 | 近一些，读者熟悉领域 | 验证框架在专业领域成立 |
| 第三个 | 当下、读者正在经历 | 框架命中现实，读者接受 |

顺序反过来效果会大打折扣。先说当下再说历史，读者还没被"训练好"就要接受结论。

---

### 3. 锚点句：一句话压缩整篇论点

每篇好文章都有一个句子，能从它展开整篇，也能把整篇压回它。

**特征：**
- 不是总结，是浓缩
- 读者能记住，能转述
- 通常出现在中段，不是结尾

**示例：**
> "让你的判断力变得机器可读"

**写法：** 写完初稿后问自己："如果只留一句话，是哪句？" 然后把这句话打磨到最精准。

---

### 4. 先共情常见误判，再纠正

直接说"你错了"会触发防御。先复述读者自己说过的话，再温和地给出正确因果链。

**结构：**
```
"[读者常说的话]" ← 引用，制造认同感
这个诊断几乎总是错的。← 否定
真正的原因是…… ← 给出正确框架
```

**效果：** 读者不抵触，因为感到被理解，而不是被批评。

---

### 5. 结尾给身份重定位，不给任务清单

"所以你应该做 X、Y、Z" → 读者看完就忘。

"你是谁"的重新定位 → 读者会一直记得。

**做法：** 结尾不给 to-do list，给一个**身份锚点**——读者在新世界里是什么角色。

**示例：**
> "那些工人再也没有回去拧阀门。不是因为做不到，而是因为那已经没有意义了。"

读者想的不是"我要做什么"，而是"我是谁"——触动更深，更持久。

---

### 6. 点出生成-验证不对称性

当文章涉及人机协作、AI 工具、自动化时，用这个结构帮读者重新定位自身价值：

> 生成一个正确的解比验证一个解要难。
> 你不需要在**实现**能力上超越机器，你需要在**评判**能力上超越它。

这让读者从"我会被取代吗"转向"我的核心价值在哪"。

---

## Quick Reference

| 要做 | 不要做 |
|------|--------|
| 框架先行，事实服从框架 | 堆砌信息，让读者自己归纳 |
| 例子由远及近，逐步扩射 | 直接从当下例子开始 |
| 找到一句锚点句并打磨它 | 用多个结论稀释核心论点 |
| 先引用误判再温和纠正 | 直接说"你错了" |
| 结尾给身份重定位 | 结尾给 to-do list |
| 指出新角色的价值所在 | 只描述变化，不给出路 |

## Common Mistakes

**框架太宽泛：** "这是一个关于变化的故事" → 没有解释力，换成具体的历史概念或理论名词

**例子太同质：** 三个例子都来自软件行业 → 跨度不够，框架的普适性无法被验证

**锚点句太长：** 超过 20 字就记不住 → 反复压缩，直到可以一口气说完

**结尾说教：** "所以我们都应该……" → 读者感到被教训，改成一个具体场景或历史事实

**共情不够就纠正：** 直接批评常见做法而不先引用它 → 读者防御，关闭接受通道
